# 08 — Admin панел (CMS) спецификација

## 1. Цел

Прост, брз, безбеден internal alat за Гјоргји (и опционо Тамара) да:
1. Пишува, уредува, објавува и брише blog постови — без потреба од WordPress admin или
   надворешен CMS.
2. Прегледува, означува статус, и одговара на пораки од контакт формата.

**Не** е јавен/multi-tenant систем — само 1-2 internal корисници. Дизајнот треба да биде
функционален и брз пред да биде "убав" (за разлика од јавниот сајт каде визуелниот идентитет е
приоритет).

## 2. Автентикација

- Laravel Breeze (Inertia/Vue стек) — login форма на `/admin/login` (одделена визуелно од јавен
  сајт, видете `02-TECH-STACK.md` §7).
- Корисници се креираат **рачно** преку `php artisan tinker` или Database seeder, нема јавна
  registration рута:

```php
// database/seeders/AdminUserSeeder.php
User::create([
    'name' => 'Гјоргји',
    'email' => 'gjorgji@financebuddy.mk',
    'password' => Hash::make('***'), // силна лозинка, сменета веднаш по прв login
    'role' => 'admin',
]);
```

- Опционо за подобрена безбедност (не задолжително за MVP, но препорачано бидејќи admin панелот
  контролира јавна содржина): 2FA преку `laravel/fortify` пакет, или едноставно ограничување на
  admin login само од познати IP адреси/преку VPN ако Гјоргји секогаш пристапува од истите
  локации — последново е премногу рестриктивно за реален употреба сценарио, па препорака е само
  силна лозинка + rate limiting на login обиди (видете `04-LOGGING-MONITORING.md` §2.2) за MVP.

## 4. Блог управување

### 4.1 Листа постови (`/admin/blog`)

Табела: наслов, категорија, статус (draft/published бедж), датум на објава, акции (уреди/
избриши/preview). Филтри: по статус, по категорија. Sort: по датум (default), по наслов.

### 4.2 Нов / уреди пост (`/admin/blog/novo`, `/admin/blog/{post}/uredi`)

```
Поле                 Тип                    Забелешка
─────────────────────────────────────────────────────────────────
Наслов                text                   автоматски генерира slug (уредлив рачно)
Slug                  text                   editable, со real-time превид на финалната URL
Категорија             select                 dropdown од постоечки + "Нова категорија" опција
Кратко резиме          textarea (кратко)      користи се за preview картички + meta description fallback
Главна слика           file upload            preview thumbnail, валидација MIME+големина
Содржина               Markdown editor         live preview split-view (видете §4.3)
─── SEO секција (collapsible, не задолжителна да биде отворена секогаш) ───
Meta наслов             text (optional)         fallback: користи "Наслов" ако празно
Meta опис               textarea (optional)      fallback: користи "Кратко резиме" ако празно
─────────────────────────────────────────────────────────────────
Статус                  toggle/select           Нацрт (draft) / Објавено (published)
[Зачувај нацрт]  [Прегледај]  [Објави]
```

### 4.3 Markdown editor — препорака

Користи готова Vue 3 компонента (на пр. `md-editor-v3` npm пакет) наместо градење custom rich-text
editor од нула — заштедува значително време и ризик. Split-view (Markdown лево, live HTML preview
десно) е стандардна, позната UX за секого кој некогаш пишувал во Markdown (GitHub, Notion, итн.).

**Зошто Markdown, не WYSIWYG (Word-style) editor:** чува содржина чиста, спречува "messy HTML"
проблеми (copy-paste од Word носи скриени стилови), полесно за backup/migration во иднина, и е
побезбедно (помал XSS risk простор отколку rich-text HTML editor кој дозволува произволни tags).

### 4.4 Категории управување

Едноставна CRUD страница (`/admin/blog/kategorii` — опционо посебна страница, или inline
управување во "Нов пост" формата преку "Нова категорија" брз dropdown option). За MVP, inline
創ирање преку dropdown е доволно — не е потребна посебна целосна CRUD страница за нешто толку
едноставно.

### 4.5 Преглед пред објавување (Preview)

"Прегледај" копче отвора нов tab со точно како постот ќе изгледа на јавниот сајт, дури и ако
статусот е сѐ уште "draft" (auth-заштитен preview route достапен само за логиран admin корисник,
користејќи signed URL или auth check — никогаш јавно достапен draft содржина).

## 5. Контакт пораки управување

### 5.1 Листа пораки (`/admin/poraki`)

Табела/картички приказ: име, email, кратка preview на пораката, датум, статус бедж (Ново /
Прочитано / Одговорено / Архивирано). Default sort: најнови прво. Филтер: по статус.

**Визуелен сигнал:** нови (непрочитани) пораки визуелно истакнати (bold, или мала точка/badge),
слично на email inbox конвенција — позната, интуитивна UX, нема потреба да се изнаоѓа нов патерн.

### 5.2 Единечна порака (`/admin/poraki/{message}`)

Целосен приказ на пораката + метаподатоци (IP адреса, datum, кој service_interest е selektиран
ако применливо). Акции:
- Промени статус (Прочитано / Одговорено / Архивирано) — рачно, или автоматски "Прочитано" штом
  се отвори детален приглед (стандардна inbox конвенција).
- "Одговори преку email" копче — `mailto:` link со prefilled subject (брзо, без потреба од
  вграден email клиент во апликацијата — overkill за овој обем).

### 5.3 Спам заштита (видете и `03-CODING-STANDARDS.md` §8)

- Honeypot поле (скриено поле кое луѓе не го пополнуваат, но ботови автоматски да; ако е
  пополнето, пораката тивко се отфрла без error за ботот, но НЕ се зачувува во база).
- Rate limiting (throttle middleware) на submit рутата.
- Ако спам стане проблем по лансирање, разгледај Cloudflare Turnstile (privacy-friendly
  алтернатива на Google reCAPTCHA) — не е потребно за MVP, додади само ако реално стане проблем.

## 6. Дозволи (permissions) — едноставен модел

За само 1-2 корисници, целосен role-based permission систем е претерано. Едноставен модел:

```
role: 'admin'   — целосен пристап (CRUD блог, преглед/управување пораки, идно: user management)
role: 'editor'  — (опционо, ако подоцна се додаде друг писател) CRUD блог само, без пристап до пораки
```

За MVP, само `admin` role е доволна (Гјоргји + опционо Тамара, двајцата со целосен пристап).

## 7. Што НЕ влегува во MVP admin панел (свесно исклучено)

- Media library/file manager (само директен upload по пост, не централизирана галерија) — додади
  подоцна ако обемот на слики го бара тоа.
- Multi-step approval workflow (draft → review → publish) — премногу за тим од 1-2 луѓе.
- Аналитика вградена во admin (користи Google Analytics / Search Console директно наместо
  градење custom dashboard).
- Версионирање/revision history на постови — Git ја снима кодната база, но content версии во
  база не се version-controlled во MVP; ако тоа стане потребно, разгледај подоцна.

## 8. Чеклиста пред лансирање (admin)

- [ ] Admin рути целосно недостапни без login (тестирано директно преку URL, не само "нема линк
      во UI")
- [ ] Силни лозинки поставени, default/placeholder лозинки сменети
- [ ] Markdown content правилно се рендерира и санитизира на јавна страница (тест со
      намерно "лош" markdown - script tags, итн., да се потврди дека се escape-увани)
- [ ] Email известувања за нови пораки тестирани (реално стигнуваат во inbox, не во spam)
- [ ] Draft постови не се видливи на јавна страница ниту преку директна URL погодка
